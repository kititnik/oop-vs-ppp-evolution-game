# Выводы в процессе разработки
Главная проблема стадии 3 - **наложение двух паттернов**(Фабричный метод + Стратегия)

В Stage 2 уже был один паттерн Visitor для double dispatch при атаке. Здесь же стратегия `BattleStrategy` сама вынуждена диспетчеризоваться по типу юнита(Warrior ведёт себя не так как Worker), и в ООП это снова Visitor внутри Strategy. То есть паттерн поверх паттерна.

Рассмотрю стоимость расширения при добавлении нового типа `Archer`. Для этого разработчик должен:

### ООП
- реализовать `Archer.h` и `Archer.cpp`.
- добавить метод `attackArcher(Archer&)` в интерфейс `UnitAttacker`(прежняя проблема из Stage 2);
- добавить метод `actArcher(Archer&, ...)` в интерфейс `BattleStrategy`;
- реализовать `executeArcher` во всех существующих стратегиях(`RandomStrategy`, `AggressiveStrategy`);
- реализовать `attackArcher` во всех атакующих классах;

Это нарушает принцип **открытости/закрытости** дважды, в `UnitAttacker` и `BattleStrategy`.

### ППП
- в новых файлах `archer.h` и `archer.c` добавляются специализации мультиметодов:
    - `warrior_attack<Unit.warrior* attacker, Unit.archer* target>`;
    - `strategy_execute<BattleStrategy.random* s, Unit.archer* unit>`;
    - `strategy_execute<BattleStrategy.aggressive* s, Unit.archer* unit>`.

Таким образом все обработчики нового типа сосредоточены в одном файле.

С добавлением новой стратегии и в **ООП**, и в **ППП** изменения локализованы в новых файлах. Здесь подходы равны. Но при добавлении нового типа `Unit` в **ООП** нужно менять классы стратегий. А в **ППП** изменения можно локализовать в новых файлах(эту возможность в ходе разработки Stage 3 я не использовал)
