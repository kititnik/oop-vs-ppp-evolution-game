# Выводы в процессе разработки
Главная проблема стадии 2 - **double dispatch**

Рассмотрю стоимость расширения при добавлении нового типа `Archer`. Для этого разработчик должен:
### ООП и паттерн Visitor
- добавить метод `attackArcher(Archer&)` в интерфейс `UnitAttacker`
- реализовать этот метод во всех существующих классах атакующих (`Warrior`, и любых будущих);
- реализовать `acceptAttack` в самом `Archer`.

Это нарушает принцип **открытости/закрытости**, так как для добавления одного типа требуется менять базовый класс и все классы атакующих

### ППП и мультиметоды
- в новом файле расширяется обобщение `Unit + <archer: Archer;>;`
- добавляется специализация `warrior_attack<Unit.warrior* attacker, Unit.archer* target>`;

Если Archer тоже атакует, все его мультиметоды атаки сосредоточены в `archer.c`. Изменение изолировано в одном месте. `Unit`, `warrior` и другие заголовки/функции не меняются.
