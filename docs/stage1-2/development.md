# Процесс разработки
### Изменения до перехода
Для ООП некоторые поля(особенно для класса `Unit` и производных) стали `public`. Для ПП подхода все классы кроме `Unit` и его специализации переведны на использование `opaque pointer`. Всё это необходимо для синхронизации полиморфизма в обоих стилях.

### Общие
**Базовые механики:** Добавлены здоровье юнитов, ресурсы для племени, метод поиска случайной живой цели в племени. Теперь в симуляции участвуют два племени, что повлекло изменения в струткуре(классе) Simulation, а также при инициализации племён. Создана структура `SimulationContext`, которая для каждого племени хранит собственное и вражесткое племя и передаётся на каждом шаге симуляции. Реализация этих механик почти полностью **симметрична** для обоих подходов.

### ООП
**Double Dispatch (Visitor):** Для реализации атак `Warrior` применён паттерн **Visitor**. Введён интерфейс `UnitAttacker`. Все Unit получили метод `acceptAttack`, Warrior отдельно получает `attackWarrior` и `attackWorker`:

```c
warrior.act(context)
→ target = context.enemyTribe.getRandomAliveUnit()
→ target.acceptAttack(warrior)          // первый dispatch: тип цели
    → warrior.attackWarrior(target)       // второй dispatch: тип атакующего
```

### ППП
**Double Dispatch (Мультиметоды):** В отличие от ООП, механика атаки была реализована через **мультиметоды** (`warrior_attack` для конкретных пар) без дополнительных абстракций. Вызов автоматически разрешается в нужную специализацию на основе реального типа обоих аргументов во время выполнения:
```c
void warrior_attack<Unit.warrior* attacker, Unit.warrior* target>(); // Warrior vs Warrior
void warrior_attack<Unit.warrior* attacker, Unit.worker* target>();  // Warrior vs Worker
```
